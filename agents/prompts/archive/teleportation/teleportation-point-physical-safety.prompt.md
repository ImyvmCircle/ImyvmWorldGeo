# 20260221ImyvmWorldGeoTeleportationPointFix

## 项目通用上下文

1. 本项目很大程度上是一个Core，即为一系列基于区域的mod提供基本依赖。故而，本mod最大的功能是维护`Region`及其附带效果，并提供`entrypoints/api`中的接口。指令也要配套制作，指令的主要功能是紧急情况下的手工底层介入，应该要制作的比较全面为好。
2. 对以下提及的工具，不管新增什么功能，均应该优先调用，如无，则在尽可能接近的模块平行新建。
3. 本mod有i18n系统，通过`Translator.tr()`函数实现，目前实现的名称均为英文，请不要做中文文件。原则上不要使用`Text.literal()`。需实现`resource`里面对应的英文项目。对于发送给玩家的文本，需用写清楚操作的对象（特别是操作者，Region, Scope, 属性，目标玩家，操作内容等）但不要引入Unicode特殊符号。本mod的语言系统是支持MOTD下的丰富显示的，请使得显示更加丰富，带有各种醒目的颜色和提示等效果。
4. 项目的几何工具全部都在`util/geo`中。
5. 项目有三个基础设施，`lazyTicker`,`RegionDatabase`和`WorldGeoConfig`. 第一个用来注册每隔一段时间进行一次的事件，第二个是每次`Region`对象有数据被修改的话都要进行的保存操作，第三个存储了所有具体数值，任何具体数值都应该写在其中。
6. 修改完之后，要进行清晰的总结并修改`README.md`的相关模块，且前述任务中，不要使用各种emoji以及特殊Unicode符号。

## 任务通用上下文

1. 传送点是`Region`的`GeoScope`集合中的一项属性。传送点可以为null，表示没有传送点，但一般是一个坐标。传送点的其他属性，如Accessbility相关的属性，可以修改，但本mod不需要实现，是给别的mod提供的一项辅助参数。
2. 传送点最重要的要求是物理意义上对于玩家安全。这项检查不仅应该在设定意义上检查，还应该在每次执行传送的时候检查。
3. `TeleportPointApplication`里面放置了传送点相关的所有实现。

## 任务内容

1. 以你的视角，提出物理安全的一个合理补充方案，研究是否需要补充现有的物理安全检查内涵。如无，则直接告知即可，不用强行凑点。另外，在保证物理意义上安全的前提下，检查并扩展做到
   - 传送点现有的物理安全意涵是否允许传送点设置在室内；
2. 增加传送时的物理安全检查。如物理意义不安全，首先观察是否能通过在周围$5\times5\times5$的区间内以高度优先进行搜索找到一个最近的、符合传送点设置要求（包括现有的各种位置要求等，当然也包括物理安全要求）的新传送点位。如有，更新传送点位，并将玩家传送到该处。如无，则不再进行传送。
3. 在给玩家提示的消息中，在是传送点设置或者传送未达成的各种情况中，都对物理安全的内涵进行详细的界定和说明，并且给玩家指出是哪里物理意义上不安全，造成了什么未达成。
