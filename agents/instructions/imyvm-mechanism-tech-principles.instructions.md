# IMYVM 机制技术原则

1. 实现或改动资源发放、资源收取、跨模块入账或扣款机制时，区分账户型资源和物品型资源。账户型资源通过账户 API 变更余额；物品型资源通过物品发放或扣除处理。资源操作使用统一资源栈或等价抽象表达，发放与收取接口返回结构化结果；涉及账户、组织资产或外部系统资产变动时记录流水编号。账户扣款保持原子性，余额不足返回明确失败；物品扣除按物品 ID、自定义名称、lore、附魔和 NBT 严格匹配。
2. 实现或改动结算机制时，将事实冻结与资源转移分为两个阶段。冻结阶段确定主体、数量、来源、周期等事实并写入不可变结算记录；执行阶段只调用资源或账户写入。执行失败时保留待处理状态，重试沿用冻结事实，不重新计算、不重新随机。
3. 实现或改动任何金钱或资源的发放、收取、结算、补发、扣款机制时，必须显式处理玩家在线与离线两种状态，不得只按在线玩家路径设计。账户型资源发放可在离线时写入待入账队列，并在玩家登录或重试任务中执行；账户型资源收取仅在业务明确允许离线扣款时进入待处理记录。物品型资源在线优先进入背包，背包满或玩家离线时进入待领取队列，并按业务规则设置过期与过期后处理；默认不对离线玩家扣除物品。玩家登录时应汇总提示可领取、待入账或待处理事项，并提供查询入口。
4. 实现或改动失败重试、待处理队列或补偿机制时，待处理记录必须保存周期编号、主体、资源、金额或数量、来源、外部引用、失败阶段、失败原因、重试次数和下次重试时间。重试入口至少覆盖下一结算周期、玩家登录和 OP 手动触发；重试必须沿用原冻结事实与随机结果。待处理队列必须持久化并在服务器重启后恢复，不得因重试重复结算同一周期；失败原因必须分类，至少区分余额不足、玩家离线和外部 API 异常。
5. 实现或改动周期结算、定时结算或周结束任务时，所有结算记录必须携带周期编号与交易流水编号，用于对账、幂等和重复执行防护。周期边界以 WorldGeo 时间服务为准，默认按东八区计算；服务器启动时应补算错过的周期，但不得重复结算同一周期。周结束行为应纳入统一任务管理，包括防刷热度重置、占领结算、税收福利结算或同类周期性机制。
6. 实现或改动玩家交互、菜单 lore、提示、公告、日志下行或结算反馈消息时，必须先与用户确认消息分层：哪些消息所有玩家可见，哪些只对对应管理者与 OP 可见，哪些仅 OP 可见，哪些属于系统私有信息。该分层只影响信息发送，不改变业务权限判断。系统私有参数只允许存在于服务端持久化、配置或内部计算中，不得出现在任何下行消息、菜单 lore 或玩家提示里；系统私有参数包括随机种子、精确公式、风控阈值、反刷权重或同类实现细节。
7. 实现或改动跨项目机制时，下层项目不得出现上层业务语义；上层项目只能通过下层公开 API 组合能力，不直接依赖下层的 `infra`、`application` 或 `mixin` 包。与 Minecraft 原版代码的交互应集中在各项目的 `infra` 或 `entrypoint` 层；能用 Fabric API 完成时优先使用 Fabric API，只有在公开 API 不足以表达目标行为时才考虑 mixin。
